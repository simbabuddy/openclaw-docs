## 如何贡献

### 方式 1: 修正错别字或语法错误
- 直接编辑文件并提交 PR
- 小修改不需要创建 Issue

### 方式 2: 新增或重写文档
1. 先查看是否已有相关文档
2. 创建 Issue 说明你的计划
3. 等待确认后开始编写
4. 提交 PR 并等待 review

### 方式 3: 翻译文档
- 选择你要翻译的文档
- 在对应目录创建 `{lang}/` 子目录
- 提交翻译版本

## 文档规范

### 文件命名
- ✅ 使用小写字母和连字符：`getting-started.md`
- ❌ 不要用空格或大写字母：`Getting Started.md`
- ✅ 日期格式：`2026-03-02-meeting.md`

### 目录结构
```
docs/
├── public/           # 公共文档
├── roles/            # 角色文档
├── internal/         # 内部文档
└── templates/        # 模板
```

### Markdown 格式
- 使用 `#` 标题，保持层级清晰
- 代码块标注语言：\`\`\`python
- 链接使用相对路径（内部）或完整 URL（外部）
- 图片放在 `assets/` 目录

### 文档结构模板
```markdown

_简短描述_

## 概述

背景、目的、适用范围

## 提交流程

### 1. Fork 仓库
```bash
# 在 GitHub 上点击 Fork 按钮
```

### 2. Clone 到本地
```bash
git clone https://github.com/YOUR_USERNAME/openclaw-docs.git
cd openclaw-docs
```

### 3. 创建分支
```bash
git checkout -b docs/add-new-feature-guide
```

分支命名规范：
- `docs/{描述}` - 新增/修改文档
- `fix/{描述}` - 修正错误
- `translate/{语言}/{描述}` - 翻译

### 4. 编写文档
```bash
# 使用你喜欢的编辑器
code docs/public/user-guide/new-feature.md
```

### 5. 提交更改
```bash
git add docs/public/user-guide/new-feature.md
git commit -m "docs: 添加新功能使用指南"
```

Commit 信息规范：
- `docs: {描述}` - 文档相关
- `fix: {描述}` - 错误修正
- `feat: {描述}` - 新增内容

### 6. 推送并创建 PR
```bash
git push origin docs/add-new-feature-guide
```

然后在 GitHub 上创建 Pull Request。

## PR Review 流程

1. **自动检查**
   - 格式检查
   - 链接检查

2. **人工 Review**
   - 内容准确性
   - 文档结构
   - 语言表达

3. **合并**
   - Review 通过后合并到 main 分支
   - 自动部署到文档站点

### API 文档模板
```markdown
# API 名称

## 请求

```
METHOD /api/endpoint
```

### 请求头
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| Authorization | string | 是 | Bearer Token |

### 请求参数
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| param1 | string | 是 | 参数说明 |

## 响应

### 成功响应 (200)
```json
{
  "status": "success",
  "data": {}
}
```

### 错误响应
| 状态码 | 说明 |
|--------|------|
| 400 | 请求参数错误 |
| 401 | 未授权 |
| 500 | 服务器错误 |

## 示例

```bash
curl -X GET https://api.example.com/endpoint \
  -H "Authorization: Bearer YOUR_TOKEN"
```
```

## 常见问题

**Q: 我可以修改公共文档吗？**

A: 可以！但需要提交 PR 并经过 review。确保你的修改准确且有帮助。

**Q: 角色文档谁能访问？**

A: 角色文档对所有人可见，但主要由对应角色的成员维护和更新。

**Q: 内部文档如何访问？**

A: 内部文档仅限核心团队成员。需要联系管理员获取权限。

**Q: 文档更新频率？**

A: 功能变更后 1 周内应更新对应文档。定期审查和更新过时内容。
